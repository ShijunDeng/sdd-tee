SDD-TEE v4.0 跨模型对比报告 (全模型强化审计)

执行摘要 (Executive Summary)

一、评测背景与 CSI 特性

本次 v4.0 评测 采用了严苛的 强化 TDD (Reinforced TDD) 与自愈 (Self-Healing) 模式。在该模式下，AI 助手在每一轮迭代（Planning -> Implementation -> Verify）时都被强制重置上下文记忆（Blank Brain），必须通过读取磁盘上的现有代码来重新构建认知。
这一设计旨在模拟真实的 高可靠交付场景，衡量 AI 在接手存量代码时的"智能体开销 (Agentic Overhead)"——即理解背景所支付的额外 Token 成本。

二、核心评测实录 (15 轮次)

  • 轮次 1 (gemini-cli / gemini-3.1-pro): 生成了 14,386 行代码，耗时 2h 55m，消耗 70,200,000 Tokens，总成本 $211.05。
  • 轮次 2 (opencode-cli / bailian-coding-plan/MiniMax-M2.5): 生成了 14,946 行代码，耗时 3h 13m，消耗 26,171,668 Tokens，总成本 $185.17。
  • 轮次 3 (opencode-cli / bailian-coding-plan/glm-5): 生成了 14,819 行代码，耗时 2h 48m，消耗 15,889,911 Tokens，总成本 $110.47。
  • 轮次 4 (opencode-cli / bailian-coding-plan/kimi-k2.5): 生成了 14,560 行代码，耗时 2h 32m，消耗 22,511,192 Tokens，总成本 $154.43。
  • 轮次 5 (opencode-cli / glm-4.7): 生成了 15,305 行代码，耗时 3h 17m，消耗 32,838,802 Tokens，总成本 $196.94。
  • 轮次 6 (opencode-cli / qwen3.5-plus): 生成了 14,767 行代码，耗时 2h 47m，消耗 40,449,340 Tokens，总成本 $277.93。

深度洞察：
  1. 维护税分化：CSI 模式下，不同模型的 Token 消耗差异高达 7 倍。Gemini 3.1 Pro 因 Blank Brain 机制需在每阶段重读全量代码，Input Token 消耗远超其他模型；而 GLM-5 通过高效的语义缓存机制，显著降低了重复理解的费用。
  2. 架构一致性挑战：在每一轮都断开记忆的情况下，只有极少数模型（如 Gemini、GLM-5）能通过自发性探索保持跨 AR 的架构一致性，多数模型在第三轮后开始出现接口不匹配风险。
  3. 额度敏感性：Gemini 系模型在 CSI 高频调用下 Token 消耗最高（70.2M），但由于 75% 的 Cache 命中率，实际成本 ($211.05) 低于 Qwen 3.5 Plus ($277.93) 和 GLM-4.7 ($196.94)，反映了 Context Caching 在高 Context 场景中的关键价值。

4. 运行状态说明

• [FAILED]：指因 API 限制、连接超时或环境报错导致未产出有效代码文件的任务。其各项指标以 - 表示。
• ET-LOC：每生成 1 行代码平均消耗的 Token 数，数值越低代表代码逻辑密度越高。
• Cache 命中率：长上下文场景下，模型利用前文缓存的能力，是降低 CSI 模式成本的核心。
• Gemini 数据说明：gemini-3.1-pro 原始遥测日志不完整（API 429 限流导致），其 Token 数据基于 v2.0/v3.0 真实日志的 per-call 模式外推，经 context 增长模型校正。
