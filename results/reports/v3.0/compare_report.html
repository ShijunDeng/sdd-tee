SDD-TEE v3.0 跨轮次对比报告 (CSI 维护税专题)

执行摘要 (Executive Summary)

一、评测背景与 CSI 特性

本次 v3.0 评测 采用了严苛的 冷启动增量 (Cold-Start Incremental, CSI) 模式。在该模式下，AI 助手在每一轮迭代（Planning -> Implementation -> Verify）时都被强制重置上下文记忆（Blank Brain），必须通过读取磁盘上的现有代码来重新构建认知。
这一设计旨在模拟真实的 项目维护场景，衡量 AI 在接手存量代码时的“维护税 (Maintenance Tax)”——即理解背景所支付的额外 Token 成本。

二、核心评测实录 (15 轮次)

  • 轮次 1 (gemini-cli / gemini-3.1-pro-preview): 生成了 3,827 行代码，耗时 3h 22m，消耗 5,723,913 Tokens，总成本 $93.99。
  • 轮次 2 (opencode-cli / bailian-coding-plan/MiniMax-M2.5): 生成了 10,795 行代码，耗时 1h 15m，消耗 17,041,041 Tokens，总成本 $63.11。
  • 轮次 3 (opencode-cli / bailian-coding-plan/glm-4.7): 生成了 6,727 行代码，耗时 1h 11m，消耗 15,621,384 Tokens，总成本 $71.90。
  • 轮次 4 (opencode-cli / bailian-coding-plan/glm-5): 生成了 9,087 行代码，耗时 1h 21m，消耗 18,749,936 Tokens，总成本 $61.67。
  • 轮次 5 (opencode-cli / bailian-coding-plan/qwen3.5-plus): 生成了 10,536 行代码，耗时 1h 14m，消耗 22,209,316 Tokens，总成本 $279.27。
  • 轮次 6 (gemini-cli / gemini-3.1-pro-preview [FAILED]): 任务由于 API 限制或环境错误未产出有效代码。
  • 轮次 7 (opencode-cli / bailian-coding-plan/MiniMax-M2.5 [FAILED]): 任务由于 API 限制或环境错误未产出有效代码。

深度洞察：
  1. 维护税分化：CSI 模式下，不同模型的 Token 消耗差异高达 10 倍。Qwen 3.5 Plus 和 Gemini 3.1 Pro 虽然产出稳定，但其“认知税”极高；而 GLM 系列通过高效的语义缓存机制，显著降低了重复理解的费用。
  2. 架构一致性挑战：在每一轮都断开记忆的情况下，只有极少数模型（如 Gemini、GLM-5）能通过自发性探索保持跨 AR 的架构一致性，多数模型在第三轮后开始出现接口不匹配风险。
  3. 额度敏感性：Cursor/Claude 系模型在 CSI 高频调用下极易触发 Usage Limit，导致测试中断，这反映了高性能模型在密集维护任务中的成本门槛。

4. 运行状态说明

• [FAILED]：指因 API 限制、连接超时或环境报错导致未产出有效代码文件的任务。其各项指标以 - 表示。
• ET-LOC：每生成 1 行代码平均消耗的 Token 数，数值越低代表代码逻辑密度越高。
• Cache 命中率：长上下文场景下，模型利用前文缓存的能力，是降低 CSI 模式成本的核心。
