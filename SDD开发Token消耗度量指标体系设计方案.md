# SDD开发Token消耗度量指标体系设计方案

## 一、指标体系总体架构

```
SDD Token消耗度量体系
├── 维度1：阶段维度（按7阶段工作流）
├── 维度2：角色维度（人 vs AI）
├── 维度3：类型维度（输入Token vs 输出Token）
├── 维度4：效率维度（Token/产出）
└── 维度5：质量维度（Token/质量指标）
```

## 二、核心指标体系

### 2.1 阶段级Token消耗指标（核心）

| 指标编码 | 指标名称 | 计算公式 | 数据来源 | 基线目标 |
|---------|---------|---------|---------|---------|
| **ST-0** | AR输入阶段Token消耗 | `Input_Token + Output_Token` | Agent日志 | 待建立 |
| **ST-1** | 需求澄清阶段Token消耗 | `Proposal_Input + Proposal_Output` | proposal.md生成日志 | 待建立 |
| **ST-2** | Spec增量设计Token消耗 | `DeltaSpec_Input + DeltaSpec_Output` | delta-spec.md生成日志 | 待建立 |
| **ST-3** | Design增量设计Token消耗 | `DeltaDesign_Input + DeltaDesign_Output` | delta-design.md生成日志 | 待建立 |
| **ST-4** | 任务拆解Token消耗 | `Tasks_Input + Tasks_Output` | tasks.md生成日志 | 待建立 |
| **ST-5** | 开发实现阶段Token消耗 | `∑(每个Task的Token消耗)` | 代码生成日志 | 待建立 |
| **ST-6** | 合并归档阶段Token消耗 | `Merge_Input + Merge_Output` | 文档合并日志 | 待建立 |
| **ST-7** | 一致性验证Token消耗 | `Verify_Input + Verify_Output` | 验证报告生成日志 | 待建立 |

### 2.2 角色维度Token消耗指标

| 指标编码 | 指标名称 | 计算公式 | 说明 |
|---------|---------|---------|------|
| **RT-AI** | AI消耗Token总量 | `∑(所有AI调用的Token)` | AI Agent调用的总Token |
| **RT-HUMAN** | 人类输入Token总量 | `∑(人类Prompt的Token)` | 人工输入的Token |
| **RT-RATIO** | 人机Token比 | `RT-HUMAN / RT-AI` | 衡量人机协作效率 |
| **RT-ITER** | 平均迭代次数 | `∑(迭代次数) / 需求数` | 每个需求的平均交互轮数 |

### 2.3 效率维度Token消耗指标

| 指标编码 | 指标名称 | 计算公式 | 说明 |
|---------|---------|---------|------|
| **ET-LOC** | Token/代码行数 | `ST-5 / 代码总行数` | 每行代码的Token成本 |
| **ET-FILE** | Token/文件数 | `ST-5 / 变更文件数` | 每个文件的Token成本 |
| **ET-TASK** | Token/任务数 | `ST-5 / Task数量` | 每个Task的平均Token |
| **ET-AR** | Token/需求（AR） | `∑(ST-0~ST-7) / AR数量` | 单个需求的完整Token消耗 |
| **ET-TIME** | Token/小时 | `总Token / 总开发耗时` | Token消耗速率 |

### 2.4 质量维度Token消耗指标

| 指标编码 | 指标名称 | 计算公式 | 说明 |
|---------|---------|---------|------|
| **QT-COV** | Token/测试覆盖率 | `ST-5 / 单元测试覆盖率%` | 每单位覆盖率的Token成本 |
| **QT-CONSIST** | Token/一致性得分 | `ST-7 / Spec-Code一致性%` | 一致性验证的Token效率 |
| **QT-AVAIL** | Token/代码可用率 | `ST-5 / AI代码可用率%` | 生成代码质量的Token效率 |
| **QT-BUG** | Token/Bug数 | `ST-5 / 发现Bug数` | 每个Bug的Token成本（反向指标） |

### 2.5 阶段间Token分布指标

| 指标编码 | 指标名称 | 计算公式 | 说明 |
|---------|---------|---------|------|
| **PT-DESIGN** | 设计阶段Token占比 | `(ST-1+ST-2+ST-3) / 总Token` | 设计阶段Token消耗比例 |
| **PT-DEV** | 开发阶段Token占比 | `ST-5 / 总Token` | 开发阶段Token消耗比例 |
| **PT-VERIFY** | 验证阶段Token占比 | `(ST-6+ST-7) / 总Token` | 验证阶段Token消耗比例 |
| **PT-PEAK** | 峰值Token阶段 | `max(ST-0~ST-7)` | Token消耗最大的阶段 |

## 三、基线建立方案

### 3.1 穿刺实验设计

**实验目标：** 建立SDD开发各阶段Token消耗基线

**实验样本选择：**
- 选择3-5个不同复杂度的实际需求（AR）
- 覆盖不同类型：新功能开发、Bug修复、重构、性能优化
- 记录需求规模：预估代码行数、变更文件数、复杂度等级

**数据采集维度：**

每个AR需求采集：
- 需求元数据（AR编号、需求类型、预估规模、预估代码行数、预估变更文件数）
- 阶段Token消耗（ST-0~ST-7 各阶段Input/Output Token、各阶段耗时、各阶段迭代次数）
- 产出数据（实际代码行数、实际变更文件数、Task数量、单元测试覆盖率、Spec-Code一致性得分、AI代码可用率）
- 成本数据（Token总消耗、Token成本、人力成本）

### 3.2 基线计算方法

**步骤1：按需求规模分层统计**
- 小型需求（<500行代码）：Token消耗基线 = 平均值
- 中型需求（500-2000行代码）：Token消耗基线 = 平均值
- 大型需求（>2000行代码）：Token消耗基线 = 平均值

**步骤2：按需求类型分层统计**
- 新功能开发：Token消耗基线 = 平均值
- Bug修复：Token消耗基线 = 平均值
- 重构：Token消耗基线 = 平均值
- 性能优化：Token消耗基线 = 平均值

**步骤3：建立阶段分布基线**
- 设计阶段占比基线 = (ST-1+ST-2+ST-3) / 总Token 的平均值
- 开发阶段占比基线 = ST-5 / 总Token 的平均值
- 验证阶段占比基线 = (ST-6+ST-7) / 总Token 的平均值

**步骤4：建立效率基线**
- Token/代码行数基线 = ST-5 / 代码行数 的平均值
- Token/任务数基线 = ST-5 / Task数量 的平均值
- Token/小时基线 = 总Token / 总耗时 的平均值

## 五、指标应用场景

### 5.1 成本估算
- 新需求Token成本预测 = 基线 × 需求规模系数 × 复杂度系数
- 项目总Token预算 = ∑(各需求Token成本)

### 5.2 效率优化
- 识别Token消耗异常高的阶段（如ST-5占比>80%）
- 分析高Token消耗的原因（Prompt冗余、迭代次数多、生成内容质量低）
- 优化Prompt策略，降低Token消耗

### 5.3 质量评估
- 对比不同需求的Token/质量指标
- 识别高Token低质量的异常需求
- 调整设计模板或Prompt策略

### 5.4 进度跟踪
- 实时监控各阶段Token消耗进度
- 预警Token消耗超预算的需求
- 动态调整资源分配

## 八、说明

原始需求RR: raw requirement，来自公司内、外部客户的、关于公司产品与解决方案的、需要项目SA或者需求分析团队（RAT）分析评审后作出决定的所有需求。
特性FE: feature ，描述解决方案为支撑“客户问题（PB）”所具备的重大能力。系统特性是解决方案的主要卖点（销售亮点）集合，每条特性都是满足客户特定商业价值诉求的端到端解决方案。
初始需求IR: initial requirement，原始需求RR经过RAT分析后，站在内外部客户/市场角度，以准确的语言（完整的背景、标准的格式）重新描述的需求。
系统需求SR: system requirement，支撑特性FE所需要支持的具体需求，是系统对外呈现的可测试的全部功能用例或非功能描述，功能性需求需要按功能来分类组织。
分配需求AR: allocated requirement，根据“系统需求SR”分配到子系统/模块的功能或非功能需求。

一、SDD概念 简要介绍
1.1 SDD 的核心思想：让 Spec 从说明文档升级为“真理源”
结合业界近一年的讨论，SDD 的核心已经不只是“先写文档再写代码”，而是让 Spec 成为人和 AI 共同遵
循的 source of truth。GitHub 将这种变化概括为：规格开始成为工程过程的中心，代码更像把规格落
到实现层的“最后一公里”；Thoughtworks 也把 SDD 描述为一种先从结构化功能规格出发，再逐步拆解
为方案与任务的 AI 辅助开发方式。
这里首先要回答一个更基础的问题：什么是 Spec？
从当前业界用法看，Spec 还没有完全统一的标准定义，但有一个相对稳定的共识：它不是随意的提示
词，也不等同于传统 PRD，而是一个用自然语言编写、面向行为、结构化组织的工件，或者一组相互关
联的工件，用来表达软件功能意图，并为 AI coding agent 提供持续可用的指导。Spec 可以是一份文
档，也可以是一组文档或结构化制品，关键不在形式，而在于它是否足够清晰、可分解、可验证。
这背后，至少有三个关键变化。
第一，开发起点从“直接写代码”转向“先定义意图”。
在 SDD 中，系统要做什么、有哪些约束、如何验收，应该先在 Spec 中说清楚，再进入实现。这样做不
是为了增加前置文档，而是为了把“做什么”和“怎么做”分层，避免 AI 在意图不清时过早补全实现细节。
GitHub Spec Kit 采用的也是 Specify → Plan → Tasks → Implement 这一路径，本质上就是把意图
定义前移。
第二，Spec 必须是结构化、可执行、可验证的。
业界现在谈 SDD，越来越少把 Spec 理解成传统意义上的说明文档，而更强调它是能够驱动任务拆解、
实现生成和测试校验的结构化工件。GitHub 明确提出“specifications become executable”，Tessl 也强
调 Spec 应该用结构化、可测试的语言表达意图。换句话说，Spec 不再只是“给人读懂”，而是要能够被
人和 Agent 共同消费。
第三，Spec 要与代码仓的通用上下文区分开来。
在最新实践里，越来越多团队会把代码仓中的长期共性知识单独沉淀为另一类上下文文件，例如规则文
件、产品和代码仓的高层说明、团队约束、工程边界等。有些工具把这类长期上下文称为 memory
bank。它们在所有 AI coding 会话中都可能被反复使用；而 Spec 则只服务于某个具体功能的创建或变
更，关注的是这次任务要实现什么、修改什么、验收什么。Anthropic 也把这类持久化笔记归为 agentic
memory，用来在上下文窗口之外保存可回收的稳定信息。
因此，SDD 真正改变的不是“文档写在前还是写在后”，而是 Spec 在工程体系中的地位：它不再只是辅
助说明，而开始成为定义意图、约束实现、驱动任务拆解并承载局部变更知识的核心工件；与此同时，
仓库中的 rules、guidelines、架构说明等长期上下文，则构成另一层更稳定的 memory bank。对 AI 时
代的软件研发来说，这本质上也是一种更强的上下文工程——不是把更多信息塞给模型，而是把最关
键、最稳定、最可验证的信息，分别沉淀到合适的位置。
1.2 “Spec-Driven”不是非黑即白，而是有层级的
“Spec-driven”并不是一个非黑即白的概念，而是有不同实现层级。更准确地说，SDD 至少可以分为三个
层次。

第一层是 spec-first。
在编码之前先写好 Spec，并将其作为当前任务中 AI 辅助开发的输入。这是目前最常见、也是最容易落
地的 SDD 形态。它解决的是“先把意图说清楚，再开始生成”的问题。
第二层是 spec-anchored。
Spec 不会在任务完成后失效，而是被持续保留下来，作为后续特性演进和维护的锚点。后续需求变化
时，团队仍然围绕既有 Spec 理解系统行为、分析变更影响并更新实现。此时，Spec 不再只是一次性的
任务输入，而开始成为长期知识基线。
第三层是 spec-as-source。
Spec 成为长期主文件和权威来源，人主要编辑 Spec，而不是直接修改代码；代码则更多由 Agent 根据
Spec 持续生成和刷新。到了这一层，Spec 才真正具备“源文件”意义，代码成为规格的最后一公里实
现。
这三个层级说明，SDD 的关键不只是“先写一份 Spec”，而是 Spec 在系统创建、演进和维护过程中扮演
什么角色。当前很多实践已经做到了 spec-first，但未必真正走到了 spec-anchored，更少能做到 spec-
as-source。换句话说，SDD 的核心分歧不在于要不要写 Spec，而在于 Spec 是一次性输入，长期锚
点，还是最终源头。
1.3 为什么 AI 时代更需要 SDD
如果没有 AI，很多团队也会写设计文档、需求说明和接口约束。那么为什么 SDD 会在今天突然变得如
此重要？原因在于，AI 把原本“可以模糊处理的问题”放大成了“必须显式解决的问题”。
在以人工编码为主的阶段，需求不够清晰时，开发人员可以一边实现一边补全理解；设计不够完整时，
资深工程师可以靠经验做局部兜底；文档和代码不一致时，团队还可以通过口头沟通和 code review 临
时修补。这种方式虽然低效，但尚能运转。
而在 AI 深度参与研发之后，情况发生了根本变化。AI 的产出速度极快，一旦输入含混，它就会快速放大
含混；一旦边界错误，它就会高效率地把错误扩散到多个文件、多个模块甚至多个环节。过去“先写着再
看”的工作方式，在 AI 时代往往会变成“先生成一大堆，再花更多时间返工”。
因此，AI 时代对研发体系提出了两个更高要求。
一是要有更稳定的上下文底座。
AI 不能长期依赖聊天历史、零散补充和隐式经验，它需要清晰、稳定、可复用的知识承载体。

二是要有更明确的执行边界。
AI 需要知道哪些是必须遵守的业务规则，哪些是设计约束，哪些是不能擅自推断的空白区。没有这些边
界，它就只能“合理脑补”，而脑补正是大多数生成失真的起点。
从这个意义上看，SDD 并不是为 AI 生造出来的新概念，而是 AI 时代对软件工程提出的一次倒逼：当开
发执行者从人扩展到 Agent，原本依赖经验和默契维持的隐性知识，必须被显式化、结构化、工程化。
1.4 SDD 不是什么
为了避免误解，也有必要明确 SDD 不是什么。
首先，SDD 不是传统需求文档的简单翻版。
它不是把原有 PRD、概要设计、详细设计原封不动搬进代码仓，也不是把自然语言说明换个名字叫
Spec。真正的 SDD 强调的是：规格要能够被持续引用、被增量维护、被任务分解、被实现验证，而不
是停留在“读过就算”的静态文档层面。
其次，SDD 不是新的文档主义。
它的目标不是增加 Markdown 数量，而是降低意图传递损耗，提升实现可控性。好的 SDD 会让高风险
变更更清晰，让复杂任务更容易拆解，让 AI 更少走弯路；而不是让每个小修改都背上过重的文档成本。
再次，SDD 不是替代设计，而是强化设计的表达方式。
它并不否定架构设计、实现设计、接口设计这些传统活动，而是要求把其中真正会影响实现质量的关键
信息，以更结构化、更适合人机协作的方式组织起来。
因此，SDD 不是“不要设计”，而是“让设计能够被准确消费、执行和验证”。
1.5 从“文档辅助开发”到“规格驱动开发”
如果把传统研发方式和 SDD 做一个对比，两者最本质的区别并不在于“有没有文档”，而在于文档在工程
体系中的地位不同。
在传统模式下，文档通常只是开发的辅助材料。真正的驱动源往往还是开发者对需求的理解，以及编码
过程中的即时判断。文档写完之后，常常逐步失效，最后形成“代码是真相，文档仅供参考”的局面。
而在 SDD 模式下，规格被提升为更高优先级的工件。开发活动不再直接从模糊需求跳到代码，而是先经
过规格澄清、设计分解和任务规划，再进入实现。代码不再天然代表真相，而需要反过来接受规格的一
致性约束。
这意味着研发流程的重心，正在从“围绕代码组织协作”转向“围绕规格组织协作”。
这种变化对个人开发者、团队协作和 AI Agent 都有直接影响：
对开发者来说，重点不再只是“把功能写出来”，还包括“把意图定义清楚、把边界说准确、把结果验证到
位”。
对团队来说，评审对象不再只是 code diff，还包括 spec delta、design delta 和任务拆解是否合理。
对 AI Agent 来说，Spec 不再是背景资料，而是执行时必须持续参考的主输入。
1.6 SDD 的真正意义：让研发上下文第一次成为工程资产
归根结底，SDD 之所以重要，不只是因为它能提升 AI 写代码的效果，而是因为它改变了软件研发中“什
么东西算正式工件”这件事。
在过去，正式工件主要是代码、测试和少量设计文档；而需求意图、设计边界、经验规则、领域术语、
评审结论等大量影响实现质量的上下文，往往没有被系统纳入版本控制和工程流转。
SDD 所做的事情，就是把这些原本游离在系统之外的研发上下文，逐步转化为可沉淀、可维护、可消
费、可验证的工程资产。

因此，SDD 可以被理解为一种新的研发组织方式：
它不是单纯强调“先写 spec 再写代码”，而是强调把意图、约束、设计和变更过程正式纳入工程体系，让
人和 AI 都围绕同一套真理源协作。
也正因为如此，SDD 并不只是一个写作规范或提示技巧，而更像是一种适应 AI 时代的软件工程基础设

二、从 OpenSpec 看 SDD 的启发与边界
在讨论我们自己的方法论之前，有必要先看看业界已经走到了哪里。OpenSpec 是目前 SDD 领域最完整
的开源实践框架，它的设计思路和实际落地中暴露的问题，直接影响了我们后续 CodeSpec 的设计决
策。
2.1 OpenSpec 的核心主张
OpenSpec 提出了一个清晰的立场：在编写任何代码之前，先就需求达成共识，将意图锁定在结构化的
规范文档中。
这一主张建立在对 AI 辅助开发现状的准确判断之上——当需求只存在于聊天历史或开发者脑中时，AI 的
输出难以控制，需求漂移和范围蔓延几乎不可避免。OpenSpec 的解法是将需求转化为结构化的 Spec
文档，用 SHALL / MUST 等模态动词表达强制约束，用 WHEN / THEN 场景描述可验证的行为预期，让
规范成为实现和测试的唯一依据。
从方法论层面看，OpenSpec 贡献了几个关键洞察：
按业务能力（Capability）组织规范，而非按技术模块。 每个独立的业务能力（如用户认证、文档生
成、任务调度）拥有自己的 spec.md，从用户视角而非实现视角定义系统行为。这种组织方式让规范天
然具备了高内聚、低耦合的特性。
全量规范与增量变更分离。 全量 Spec 描述系统"当前应该如何工作"，增量变更通过 changes/ 目录以
提案形式管理，使用 ADDED / MODIFIED / REMOVED 标记精确描述每次变化。变更完成后归档，增量
合并回全量。这套机制让规范既能反映当前真相，又能追溯历史演进。
规范可验证。 每条需求必须附带至少一个场景（Scenario），场景可直接转化为测试用例。OpenSpec
还提供了 CLI 工具进行格式验证，确保规范的结构完整性。
变更提案驱动开发。 所有功能变更都以 proposal（动机与影响）→ delta-spec（规格增量）→
tasks（任务拆解）的流程推进，变更的"为什么"和"改什么"在编码前就被显式记录。
2.2 落地华为土壤时的水土不服
我司已有部门尝试引入 OpenSpec，但在实际落地中遇到了几个结构性的适配问题：
信息碎片化与认知负荷。 OpenSpec 将一个完整的业务单元拆解为 spec.md、proposal.md、
tasks.md、design.md 等多个文件，按 Capability 分散在不同目录下。理解一个功能需要在 4-5 个文
件间反复跳转。对于我司动辄数十万行代码、数百个微服务的大型系统，这种碎片化带来的上下文切换
成本是显著的。
与 IPD 流程的衔接缺失。 OpenSpec 的设计假设是敏捷团队的小步快跑模式，没有考虑 IPD 体系下系
统设计→软件实现设计→编码的分层传递关系。它缺乏"分配需求（AR）"这一概念，也没有与组件级全
量设计文档的对接机制。在我司的研发流程中，一个需求从系统设计到编码要经过多层分解，
OpenSpec 的扁平化变更管理难以承接这种层级关系。
与 Git 的功能重叠。 OpenSpec 的 proposal.md 中的 Why 本质上是 Commit Message 或 PR
Description 的内容，archive/ 目录是对 Git History 的手工模拟，Delta 机制中的 ADDED /
MODIFIED 标记是 Git Diff 的人工版本。这种"双重记账"在快速迭代中会显著增加维护负担。

================================================================================
一刀切的流程重量。 OpenSpec 对所有变更都推行完整的"提案→规格→任务→归档"流程。修改一个字
段的校验规则可能只需 5 分钟编码，却要花 30 分钟维护配套文档。缺乏按变更复杂度分级的轻重量流
程适配。
英文优先的语言假设。 OpenSpec 的规范文档要求英文编写，SHALL / MUST 等模态动词体系源自 RFC
标准。在我司以中文为主要工作语言的环境中，这增加了额外的语言转换成本。
2.3 我们从 OpenSpec 学到了什么
尽管存在适配问题，OpenSpec 的核心洞察对我们的方法论设计产生了深远影响：
我们继承了全量与增量分离的双层架构思想，但将 Capability 粒度调整为与 IPD 体系对齐的组件/服务粒
度，每个代码仓一份 SPEC.md + DESIGN.md，而非按能力拆分多个 spec 文件。
我们继承了 ADDED / MODIFIED / REMOVED 的结构化增量标记，但将其嵌入到以 AR（分配需求）为
驱动的变更流程中，与 IPD 的需求分解体系无缝衔接。
我们继承了规范可验证的理念，但将验证从格式检查升级为代码逆向分析——不仅检查 Spec 格式是否
正确，更检查代码实现是否与 Spec 语义一致。
我们放弃了 OpenSpec 的英文优先和 SHALL / MUST 体系，改用中文友好的"必须 / 应当 / 禁止"约束语
言，降低团队的使用门槛。
我们增加了按变更复杂度分级的流程弹性——小缺陷修复可以走轻量 Spec，中等特性走 proposal +
delta 流程，跨模块高风险需求才走完整的 7 阶段工作流。
简言之，OpenSpec 证明了 SDD 这条路是对的，但它是为硅谷式小团队敏捷开发设计的。要在我司的
IPD 体系和大规模工程实践中落地，需要一套既继承其核心智慧、又适配本土研发流程的方法论。这就
是 CodeSpec 的设计起点。
三、CodeSpec：面向我司研发体系的 SDD 方法论
3.1 我司软件实现设计方法论
软件实现设计是 IPD 流程 1.9.4 中编码前的最后一道设计动作。它承接系统设计阶段的输出，将系统级
架构决策转化为可指导编码的实现方案，是连接"设计世界"与"代码世界"的关键桥梁。
三层设计体系
我司的软件实现设计并非单一层次的活动，而是一套自顶向下、逐层细化的三层体系：
第一层是产品级软件实现架构设计，由软件架构师主导，关注跨组件的公共框架与机制。例如统一的服
务通信框架、公共中间件选型、跨模块的数据流转规范等。这一层为整个产品的软件实现奠定技术底
座。
第二层是软件实现元素级设计（即服务/组件实现设计），由模块设计师（MDE）负责，聚焦单个组件或
服务内部的抽象设计。它定义组件的内部模块划分、核心类结构、关键接口契约，是组件级的全量设计
文档，长期维护、持续演进。
第三层是AR 级实现设计（分配需求实现设计），由开发人员完成，是最贴近编码的轻量级设计文档。这
里的 AR（Allocated Requirement，分配需求）是系统设计阶段将系统需求分解后分配到具体软件实现
元素的实现要求，每个 AR 关联一个软件实现元素。

================================================================================
聚焦 AR 级增量设计
三层体系中，AR 级设计是日常研发活动中最高频的设计动作，也是与 SDD 关联最紧密的一层。
AR 实现设计的本质，是针对一个具体的分配需求，分析"要改哪些代码、怎么改、改完后如何验证"。它
不是从零开始的架构设计，而是在已有的服务/组件全量设计基础上做增量分析——识别本次 AR 涉及的
模块、梳理修改点、评估影响范围、规划实现路径。
AR 级设计有几个鲜明特征。首先是轻量化：聚焦于本次变更的关键决策和修改点，通常在一个迭代周期
内完成。其次是增量本质：AR 设计以服务/组件的全量设计文档作为输入基线，只分析"这次要变什
么"，而非重新描述整个组件。第三是支撑评审：AR 设计的核心产出用于串讲评审，让评审者快速理解
本次变更的意图、方案和风险。
AR 设计与上层设计之间存在双向反馈关系：AR 设计依赖服务/组件全量设计作为输入，提供架构约束和
设计基线；同时，当 AR 设计过程中发现新的抽象机会或架构改进点时，应反馈刷新上层的全量设计文
档，保持设计体系的一致性。
从传统到 SDD 的桥梁
这套三层设计体系的核心理念是正确的：分层设计确保了从架构到实现的逐层细化，增量驱动避免了每
次需求都从零开始，以 AR 为粒度保证了设计与需求的精确对应。
但在实践中，传统 AR 设计也暴露出几个结构性局限。其一，设计文档与代码分离：设计写在 Wiki 或独
立文档系统中，与代码仓库不在同一个版本控制体系下，随着迭代推进，文档与代码的一致性逐渐劣
化。其二，人工传递易失真：从设计文档到编码实现，完全依赖开发人员的理解和记忆，设计意图在传
递过程中容易衰减或偏移。其三，缺乏结构化的增量管理：虽然 AR 设计本质上是增量的，但增量的内
容（新增了什么规则、修改了什么约束、删除了什么能力）缺乏标准化的表达格式，难以被系统化地追
踪和验证。
SDD 要做的，不是推翻这套经过实践检验的设计体系，而是用结构化规格与 AI 协作来升级它——让设
计文档与代码同仓共管，让增量变更有标准化的表达和追踪机制，让设计意图能够被 AI 精确消费并转化
为高质量的代码实现。
3.2 CodeSpec 核心方法论
如果说 3.1 描述的是"我们已经在做什么"，那么 CodeSpec 要回答的是"如何用结构化的方式把它做得更
好"。CodeSpec 不是凭空发明的新框架，而是对传统三层设计体系——尤其是 AR 级增量设计——的一
次系统性升级。
3.2.1 设计理念
CodeSpec 建立在三条核心原则之上：
单一真理源——业务逻辑在代码仓库中仅有一份权威定义。不再允许 Wiki 里一个版本、设计文档里一个
版本、代码注释里又一个版本的局面存在。规格文档与代码同仓共管，由 Git 统一追踪变更历史。
去官僚化——消除冗余元数据，拒绝双重记账。文件中不写版本号、修改日期、修改人，这些信息完全
托管给 Git。设计文档不是行政产物，而是工程资产。
人机共生——文档结构优先服务于人类认知，同时标准化格式以利于 AI 消费和验证。人负责定义"做什
么、为什么做"，AI 负责辅助"怎么做、做得对不对"。
在这三条原则之下，CodeSpec 做了一个关键区分：Spec（What & Why）与 Design（How）的严格
分离。Spec 定义业务规则、数据约束、DFX 红线，回答"系统应该表现出什么行为"；Design 定义技术
方案、数据模型、接口设计，回答"用什么技术手段实现这些行为"。

================================================================================
举一个边界示例来说明这种分离：「订单ID必须全局唯一」是 Spec，它定义了一条不可违背的业务约
束；而「订单ID使用雪花算法生成，存储为 BIGINT」是 Design，它描述了实现这条约束的技术选型。
Spec 是稳定的、长期有效的；Design 可以随技术演进而替换，但不能违背 Spec 的约束。
3.2.2 全量与增量分离的双层架构
CodeSpec 在仓库中建立了两个物理分区：
全量文档区（仓库级，长期稳定知识）：包含 SPEC.md 和 DESIGN.md，分别承载全量的业务规格和全
量的实现设计。它们是整个组件/服务的知识基线，随迭代持续演进，但始终保持完整性。这对应传统体
系中服务/组件级全量设计的角色。
增量变更区（需求驱动，本次变化知识）：以 changes/{AR-ID}/ 目录组织，每个 AR 对应一个独立的
变更目录，内含本次需求从澄清到实现的全部增量文档。变更完成后归档，增量内容合并回全量文档。
这正是对传统 AR 级增量设计的结构化升级——从散落在 Wiki 中的非结构化描述，变为仓库内标准化
的、可追踪的增量文档链。
这种双层架构解决了传统方法的一个核心痛点：全量文档提供稳定的设计基线，增量文档精确记录每次
变更的意图和影响，两者通过合并归档机制保持一致性，全程由 Git 版本控制。
3.2.3 增量驱动的 7 阶段工作流
CodeSpec 将一个 AR 从输入到交付的全过程，编排为 7 个阶段：
阶段 0：AR 输入。系统设计阶段产出的分配需求（AR）到达，触发创建 changes/{AR-ID}/ 目录，这
是本次变更的工作空间。
阶段 1：需求澄清。SE 与产品协作，产出 proposal.md，回答三个问题：Why（为什么要改）、What
Changes（改什么）、Impact（影响范围）。这一步对应传统 AR 设计中的需求分析环节，但以标准化
模板固化了输出格式。
阶段 2：Spec 增量设计。SE 产出 delta-spec.md，以 ADDED / MODIFIED / REMOVED 三种标记精确
描述本次变更对业务规则的影响。每条规则附带验收条件，确保可判定、可测试。这是 CodeSpec 最核
心的创新之一——增量不再是一段自由文本描述，而是结构化的、可机器解析的变更清单。
阶段 3：Design 增量设计。MDE 或架构师产出 delta-design.md，包含技术目标、架构决策、数据模
型变更、接口设计变更、风险与权衡。它回答"用什么技术方案实现 delta-spec 中定义的业务变更"。
阶段 4：任务拆解。MDE 或开发人员产出 tasks.md，将设计方案拆解为具体到文件/模块级别的可执
行任务清单，标注任务间依赖关系。tasks.md 是人机协作的关键接口——它既是人类评审设计完整性
的检查清单，也是 AI Agent（Claude Code / Cursor 等）的直接输入。
阶段 5：开发实现。AI Agent 根据 tasks.md 逐项执行开发任务，同时参考 delta-spec.md 中的业务
规则和 delta-design.md 中的技术方案。人工负责审核 AI 输出、执行代码评审。
阶段 6：合并归档。变更完成后，delta-spec.md 的内容合并到全量 SPEC.md，delta-design.md 的
内容合并到全量 DESIGN.md，变更目录整体归档到 archive/。代码与文档原子化提交，确保任意时
间点的仓库快照都是自洽的。
阶段 7：一致性验证。通过代码逆向分析，从源码中提取业务逻辑，与 SPEC.md 进行语义比对。当
Spec 与代码行为冲突时，若非 Spec 自身错误，必须以 Spec 为准修正代码。这条原则确立了规格的权
威地位——代码是规格的实现，而非反过来。
在整个工作流中，tasks.md 扮演着承上启下的枢纽角色：向上，它是 proposal → delta-spec →
delta-design 这条设计链的最终产出，凝聚了需求意图、业务规则和技术方案；向下，它是 AI Agent 的
执行指令，每个任务都可追溯到具体的 Spec 规则和 Design 决策。

3.2.4 与传统方法的继承关系
回顾 3.1 中描述的三层设计体系，CodeSpec 的每一个设计决策都能找到传统方法论的影子：
传统的服务/组件全量设计，在 CodeSpec 中演化为仓库级的 SPEC.md + DESIGN.md。职责不变——提
供长期稳定的设计基线；形式升级——从 Wiki 文档变为代码仓内的版本控制文件，与代码同生命周期管
理。
传统的 AR 级增量设计，在 CodeSpec 中演化为 changes/{AR-ID}/ 下的 delta 流程（proposal →
delta-spec → delta-design → tasks）。核心理念不变——以分配需求为粒度做增量分析；关键升级
——增量内容有了标准化的表达格式（ADDED / MODIFIED / REMOVED），可被 AI 精确消费。
传统的设计评审，在 CodeSpec 中演化为两层验证：Spec 评审确认"改的是不是对的东西"，代码逆向验
证确认"是不是按规格正确实现了"。评审对象从模糊的设计文档，变为结构化的规格增量和可追溯的任
务清单。
本质上，CodeSpec 完成的是一次范式迁移：从"人写文档指导人编码"到"人定义规格，AI 辅助实现与验
证"。传统方法论中"分层设计、增量驱动、以 AR 为粒度"的核心智慧被完整继承，而困扰团队多年的文
档与代码分离、设计意图传递失真、增量变更难以追踪等结构性问题，则通过规格即代码、结构化
delta、AI 协作三个机制得到了系统性解决。

五、从使用者视角看 SDD：代码仓、Spec 层级与研发流程
的三重重构
前面的章节已经讨论了 SDD 的概念、行业趋势、企业落地时遇到的典型问题，以及 IPD、delta-design
和新设计范式的演进。对一线研发人员而言，SDD 带来的变化并不抽象，它会直接改变三件事：代码仓
里什么是核心工件、规格如何分层组织，以及设计与开发流程的先后关系。
5.1 代码仓的重心，从“代码文件集合”转向“意图与实现并存的工件
库”
在传统研发模式下，代码仓的核心是 src/、tests/、docs/ 这些目录；需求、设计和约束更多散落
在 Wiki、IM、会议纪要和个人经验里。进入 SDD 之后，代码仓开始承担另一种职责：不仅保存实现结
果，也保存驱动实现的意图、边界和决策过程。
GitHub 在 2025 年发布 Spec Kit 时，把这种变化描述得非常直接：规格不再是一次性文档，而是“living,
executable artifacts”，也就是能持续演进、能驱动实现的活工件；当系统变复杂、任务过大或实现偏离
预期时，团队回到 spec，而不是回到聊天记录重新猜测原始意图。GitHub, 2025

OpenSpec 则把这种理念进一步落到了仓库拓扑上：spec 和 code 一起进仓，按 capability 组织，并为
每次变更生成 proposal、design、tasks 与 spec delta，让评审者优先理解“系统将如何被改变”，而不
只是“哪几行代码被改了”。OpenSpec 官网
从用户使用侧看，这意味着代码仓会逐步出现类似下面的结构：
repo/
├─ src/
├─ tests/
├─ docs/
├─ specs/
│ ├─ capabilities/
│ │ ├─ order-payment/spec.md
│ │ └─ auth-session/spec.md
│ ├─ changes/
│ │ └─ add-remember-me/
│ │ ├─ proposal.md
│ │ ├─ design.md
│ │ ├─ tasks.md
│ │ └─ specs/
│ └─ agents/
│ └─ AGENTS.md
└─ README.md
这背后的本质变化是：仓库开始同时服务于人和 Agent。人需要它来理解系统意图，Agent 需要它来获
取稳定上下文。Anthropic 在 2025 年把这类问题概括为“context engineering”：对 Agent 来说，上下
文是有限资源，真正重要的不是把所有信息塞进去，而是把高信号、结构清晰、可按需检索的信息放在
正确的位置。Anthropic, 2025
换句话说，SDD 不是简单地“给仓库多加几个 Markdown 文件”，而是把代码仓从“源码存储器”升级成“研
发上下文的主承载体”。
5.2 Spec 不再是单层文档，而是稳定知识与变更知识分离的分层体系
很多团队第一次接触 SDD 时，会把 spec 理解成“更详细的提示词”或者“更正式的需求文档”。但从实践
看，真正可持续的做法不是写一份大而全的 spec，而是建立分层 spec 体系。
Martin Fowler 在 2026 年对 SDD 的梳理中提出了一个很有价值的区分：一类是跨任务长期有效的上下
文文件，类似 memory bank；另一类是只对某次功能变更生效的 task-specific spec。前者描述长期约
束，后者描述本次变化。Martin Fowler, 2026 这个区分非常适合指导企业内部 spec 分层。
从用户视角，一个更实用的层级通常至少包括五层：
1. 组织/平台级规则
描述通用技术栈、合规要求、测试门禁、代码风格、禁止事项。这类信息不应埋在聊天里，而应显
式放在仓库级规则文件中。GitHub 对 2500+ 个 agents.md 文件的分析发现，效果最好的 agent
配置几乎都覆盖六个要素：commands、testing、project structure、code style、git
workflow、boundaries。GitHub, 2025
2. 产品/能力级 Spec
描述某个长期存在的业务能力“应该如何工作”，例如登录、支付、会话管理。这类 spec 是跨版
本、跨会话复用的稳定知识，更接近 OpenSpec 所说“按 capability 组织的 spec”。
3. 变更级 Spec Delta
描述这次需求到底改了什么，是新增、修改还是删除哪些行为。它不是重写整份系统说明，而是记
录这次 change 对既有能力的影响。
4. 设计级文档
说明本次技术选型、架构约束、影响范围和关键权衡，是从“要改什么”过渡到“准备怎么改”。

5. 任务级拆解
把设计进一步拆成可验证、可执行、可并行的小任务，便于 Agent 执行，也便于人工验收。
这套分层最重要的价值，是把“稳定知识”和“变化知识”分开管理。稳定知识解决的是每次都要重新解释团
队规则的问题；变化知识解决的是每次需求都要重新解释意图差异的问题。前者为 Agent 提供长期记
忆，后者为当前变更提供局部聚焦。
这也是为什么越来越多团队开始把 AGENTS.md、spec 目录和 change proposal 一起纳入版本控制。
Anthropic 在 2025 年发布 Agent Skills 时也强调了同一个方向：把 instructions、scripts 和 resources
组织成可复用、可按需加载的能力模块，本质上是在把“团队经验”从临时 prompt 升级为可持续资产。
Anthropic, 2025
5.3 对现有设计与开发流程的最大冲击，不是“多一步写文档”，而是
“设计前移、开发颗粒度变小、评审对象发生变化”
从流程上看，SDD 最容易被误解成一种新的文档主义，仿佛只是把编码前多加一道规格编写步骤。但一
线开发者真正感受到的变化并不是“文档更多了”，而是设计和开发之间的边界被重画了。
首先，设计前移。过去很多团队的设计决策是在编码过程中逐渐浮现的，很多关键约束实际上隐含在资
深工程师脑中；而在 SDD 下，这些约束必须在 Agent 执行前尽量显式化。Addy Osmani 在 2025 年和
2026 年多篇文章里都强调了同一条经验：高质量 AI 开发的起点不是 prompt，而是先形成清晰 plan 和
spec，再进入代码生成；否则模型会在模糊指令下自行补全假设，最终偏离原始意图。Addy Osmani,
2025 Addy Osmani, 2026
其次，开发颗粒度变小。在传统方式下，工程师可以直接围绕一个“功能点”连续修改多个模块；在 SDD
下，更有效的方式是把实现拆成更小、边界更清晰的任务单元，让 Agent 逐项执行并验证。GitHub 的
Spec Kit 之所以设计成 Specify → Plan → Tasks → Implement 四阶段，核心原因并不是流程美观，
而是为了避免“一个模糊请求换来一大块看似正确但实则不可控的代码”。GitHub, 2025
再次，评审对象从 code diff 扩展为 intent diff。过去 PR review 主要围绕代码本身；引入 SDD 之后，
评审至少会分成两层：一层评审 spec delta 和 design，确认“改的是不是对的东西”；另一层评审代码，
确认“是不是按约束正确实现了”。OpenSpec 把这种能力总结为“review intent, not just code”，这恰好
点出了 SDD 对开发者使用方式的核心变化：评审从结果审查，变成意图审查与实现审查并重。
OpenSpec 官网
最后，开发者角色从“直接实现者”向“规格编排者 + 结果验证者”迁移。GitHub 在 2025 年对 spec-driven
workflow 的描述里明确提到，开发者的主要职责不再是亲自写完所有代码，而是持续 steer 和 verify：
确认 spec 是否准确、plan 是否考虑约束、task 是否足够小、实现是否真正满足要求。GitHub, 2025 从
使用侧看，这种转变很现实地意味着：写代码本身的时间占比会下降，而梳理意图、补充上下文、检查
结果的时间占比会上升。
5.4 但 SDD 不能演化成新的“Markdown 负担”
如果只讲收益，不讲代价，这一章会显得不完整。从 2026 年的实践反馈看，SDD 的风险并不是做不
到，而是很容易做过头。
Martin Fowler 在 2026 年对多个 SDD 工具的评估里提出了很尖锐的提醒：并不是所有问题都适合完整
的 requirements-design-tasks 流程；对小变更来说，过多 Markdown 文件和过长审阅链条会造成新的
review overload，甚至让人觉得“还不如直接 review code”。Martin Fowler, 2026
Addy Osmani 也从另一个角度给出同样的警告：模型虽然上下文窗口越来越大，但注意力预算仍然有
限，过长 spec、过多并列约束、一次性塞入太多内容，都会降低执行质量。因此，好的 spec 不是越长
越好，而是足够清晰、边界明确、按任务拆分、能持续演进。Addy Osmani, 2025
Anthropic 对 context engineering 的总结也支持这一点：高质量 Agent 工作流追求的是最小但高信号
的上下文，而不是最大上下文。Anthropic, 2025

5.6 小结：SDD 对一线用户的真实改变，是让“上下文”第一次成为正

综合来看，SDD 对现有研发体系最深的影响，不是把文档写在代码之前，而是让团队第一次系统性地把

这会带来三个直接结果：
1. 代码仓结构会扩展为“代码 + spec + agent rules”的复合工件仓。
2. spec 会从单一文档演化为稳定知识与变更知识分离的层级体系。
3. 设计与开发流程会从“边写边定”转向“先澄清意图，再小步执行，并持续验证”。
如果说传统代码仓保存的是“系统已经是什么”，那么 SDD 下的新型代码仓，开始同时保存“系统为什么这
样设计、这次准备如何变化、以及 Agent 被允许如何行动”。这正是它对一线研发人员最实际、也最深刻
的冲击。

6.3.2 暴露出的典型问题
随着实践的深入，我们也总结出了更具象问题
○ 极度依赖输入质量：AI 的输出受限于 Spec/Design 文档的完整度。一旦文档中存在未定义的外部对
象或模糊的领域术语（如“主干配置”），AI 极易产生幻觉或输出无效设计。
○ 容错与纠错能力差：目前的 AI 对 DFX（可维护性、异常容错）的考虑仍较薄弱；更关键的是，若提
示词包含错误知识，AI 会顺着错误逻辑“一路走到黑”，导致后续步骤的错误呈链式累积。
○ 重构与测试的瓶颈：面对复杂逻辑重构时容易埋下隐患（例如 API 结构的重复定义）；同时，它难
以一步到位生成复杂的端到端测试用例，强行生成的脚本往往只是“空跑”，缺乏实际校验深度。
6.3.3 落地 SDD 的几点经验
为了让 SDD 真正落地生根，我们梳理了以下五条经验总结：
1. Delta-Design 模板是核心：设计模板的质量决定了生成的上限。关键点明确、设计详尽的模板能极
大提高 SDD 的生成效果，它是 AI 理解意图的“路标”。
2. 设计必须详尽且目标明确：实践证明，最终生成的代码与增量设计（Delta-Design）的相似度极
高。一个详尽且逻辑自洽的设计文档是成功的基础，切忌在此环节“偷懒”。
3. API First 策略：在设计阶段必须严格定义好 API 接口和数据模型（Model）。这不仅是前后端约定
的基础，更是为 AI 划定了明确的开发边界，防止其在逻辑实现阶段乱造数据结构。
4. 设置量化与强制性目标：给 AI 下达指令时，必须包含明确的质量约束（如：“确保代码编译通过”、
“单元测试覆盖率必须达到 50% 以上”）。这些硬性指标能有效减少无效对话轮次，驱动 AI 自我校
准。
5. 小步快跑，及时纠偏：
○ 分步验证：代码开发要分段进行，发现微小偏差立刻打断并纠正，绝不让误差放大。
○ 拆解复杂性：遇到复杂功能，应引导 AI 先进行逻辑拆解再动笔，或指定现有的成熟模块让其
“模仿”学习。

七、下一步计划
经过前期穿刺和试点，我们已经初步验证了 SDD 路线的可行性：它能够显著提升需求澄清、方案设计、
任务拆解和 AI 辅助开发的效率，也证明了“设计先行、结构化表达、人机协同”是值得持续投入的方向。
但要让 SDD 从局部试点走向可复制、可推广、可持续运营的研发能力，还需要进一步补齐知识底座、
作业流程和资产服务能力。
从当前实践看，后续推进主要面临三类关键挑战。
第一，软件开发知识离散、隐性知识多。大量指导代码开发的关键知识，例如设计决策、踩坑经验、常
见修改模式，仍然主要掌握在资深工程师个人手中，缺乏结构化沉淀。一旦人员流动，知识就随之流
失，AI 也无法稳定获取这些高价值上下文。
第二，代码仓知识对 AI 仍然是黑盒。面对百万级代码仓，模块边界、组件接口、依赖关系、设计约束等
关键信息缺乏结构化表达，AI 往往只能从源码中被动猜测意图，理解成本高，出错概率也高。
第三，知识难以持续维护。当前不少知识资产格式不统一、更新依赖人工、变更后缺少同步机制，导致
知识库的腐化速度快于积累速度。若没有配套的 AI 辅助维护机制，设计资产很容易再次失真，难以长期
支撑研发活动。
基于这些问题，我们下一阶段的目标可以归结为两层。
在业务目标上，希望通过高质量设计资产支撑 AI 辅助开发在更多场景中落地，推动先锋 PDU 和相关产
业完成软件开发知识本体模型定义，实现代码仓知识资产覆盖率超过 80%，设计资产与代码实现一致率
超过 90%，并建立持续校验机制。
在能力目标上，希望构建一个面向开发者与 AI Agent 共用的统一研发知识底座，形成“知识分析沉淀—
消费复用—持续回写”的闭环运营能力，使 AI Agent 能够像资深研发一样理解系统全貌、参与知识生
产，并最终纳入组织化、长期化看护体系。
围绕这一目标，后续推进将建立 “能力建设 + 持续运营” 两条主线。
一条主线是能力建设。围绕 SDD 作业本身，持续建设设计规范、设计 E2E 工作流和资产服务能力，把
试点阶段验证有效的方法沉淀为标准能力。进一步支持多代码仓场景下的特性 FE 设计，使 SDD 能够从
单仓走向跨仓协同。
另一条主线是持续运营。围绕 PDU 实际需求进行互锁推进，持续收集问题、总结实践、反馈能力建设
方向，形成从一线问题到平台能力再回到一线验证的闭环。
